# CTO Mobile Strategy Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Determine and implement the safest mobile path for CTO/Morgan without fragmenting the desktop setup UI or duplicating the Rust/Tauri command surface.

**Architecture:** Keep the React/Vite setup experience as the source of truth and introduce mobile through Tauri v2 first, because this repo is already a Tauri v2 app with Rust commands, Tauri plugins, and React UI. Use React Native only if a spike proves Tauri mobile cannot support the CTO-specific native surfaces or app-store constraints.

**Tech Stack:** Tauri 2.x, React 18, Vite 6, TypeScript, Rust 1.77+, iOS/Android targets generated by `tauri ios init` / `tauri android init`, existing Node E2E/intent-test harness.

---

## Discovery facts

- Current app is Tauri 2: `src-tauri/Cargo.toml` uses `tauri = { version = "2" }` and `@tauri-apps/cli` is installed.
- Current installed Tauri CLI exposes native mobile commands: `tauri android init/dev/build/run` and `tauri ios init/dev/build/run`.
- Current repo has no committed `src-tauri/gen/android` or `src-tauri/gen/apple` target yet.
- `npx tauri info` reports local mobile blockers: Xcode is not installed, and `rustc`/`cargo` are not on PATH even though `rustup` exists.
- Existing desktop app depends on local machine/Kubernetes behaviors that do not map directly to phones: Kind cluster bootstrap, local shell/process access, source credential storage, and Tauri MCP/debug automation.
- React Native current npm version observed: `0.85.2`; Capacitor current observed: `8.3.1`; Tauri CLI observed: project 2.10.1, latest npm 2.11.0.

## Recommendation

Use a **Tauri mobile-first spike** before considering React Native.

Why:

1. The repo already owns a Tauri/Rust command layer. React Native would require rewriting native command bridges and auth/secret plumbing.
2. The Morgan setup UI is React/Vite and can be made responsive without a framework rewrite.
3. Tauri v2 has first-class `android` and `ios` commands available in the installed CLI.
4. The biggest mobile risks are product/OS constraints, not UI framework constraints: running Kind, invoking CLIs, storing secrets, background processes, and local network access.

React Native remains the fallback if the Tauri mobile spike fails on native plugin availability, mobile build reliability, or App Store/Play Store constraints that require platform-native UX.

## Target mobile product shape

Do **not** attempt to run the full desktop-local CTO stack on-device in v1.

Recommended v1 lanes:

1. **Mobile companion:** Morgan chat/status, setup review, approvals, notifications, and remote control of a desktop/cluster CTO installation.
2. **Setup assistant:** provider/source credential decision trees with QR/deep-link handoff to desktop or hosted control plane.
3. **Lightweight auth broker:** OAuth/device-code and token handoff flows, with secrets redacted and stored only in platform secure storage.

Avoid for v1:

- Running Kind/Kubernetes inside the mobile app.
- Running arbitrary local CLIs on iOS/Android.
- Assuming Tauri MCP desktop automation works on mobile WebViews.

## Task 2: Make setup UI responsive using existing React/CSS

**Objective:** Ensure Morgan setup decision-tree screens fit phone widths before any native mobile target is generated.

**Files:**
- Modify: `ui/src/styles/bootstrap.css`
- Test: add/update a static test under `scripts/e2e/` if responsive CSS conventions become machine-checkable.

## Task 3: Generate Tauri mobile targets in a controlled spike branch

**Objective:** Check whether Tauri mobile scaffolding works without committing large generated files until reviewed.

**Files:**
- Generated by spike only: `src-tauri/gen/android/**`, `src-tauri/gen/apple/**` or Tauri v2 equivalent generated paths.
- Update: `.gitignore` only if generated build artifacts need exclusion.

**Steps:**
1. Ensure Rust is available through rustup/cargo on PATH.
2. Install Xcode/Android prerequisites outside repo as needed.
3. Run `npx tauri android init`.
4. Run `npx tauri ios init` on a machine with Xcode installed.
5. Do not commit generated targets until the diff is reviewed for secrets, signing IDs, and bundle identifiers.
6. Record generated paths and blockers in `docs/2026-04/mobile-capability-inventory.md`.

## Task 4: Spike mobile command compatibility

**Objective:** Determine which Tauri commands/plugins compile and behave on iOS/Android.

**Files:**
- Modify: `src-tauri/src/lib.rs` only if conditional compilation is needed.
- Modify: `src-tauri/capabilities/default.json` or mobile-specific capabilities if generated.

**Steps:**
1. Build Android debug: `npx tauri android build`.
2. Build iOS debug: `npx tauri ios build`.
3. If shell/process/fs plugins fail on mobile, gate desktop-only commands with `#[cfg(desktop)]` or platform-specific command registration.
4. Keep source/provisioning code paths shared where possible.
5. Record every unsupported command and replacement plan.

## Task 5: Decide Tauri vs React Native

**Objective:** Make a documented go/no-go decision after the spike.

**Files:**
- Create: `docs/2026-04/mobile-framework-decision.md`

**Decision criteria:**
- Can the Tauri mobile target build on at least one simulator/emulator?
- Can the Morgan setup UI run and navigate on mobile WebView?
- Can secure storage/OAuth handoff be implemented without leaking secrets?
- Can desktop-only local stack functions be hidden or remote-routed cleanly?
- Is the generated native target maintainable by the current team?

**Default decision:** If the above pass, continue Tauri mobile. If two or more fail because of framework limits rather than missing local SDKs, plan a React Native/Expo proof-of-concept.
